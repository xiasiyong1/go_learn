# 006. Go GC 基础 - 面试追问

## 追问与参考答案

### 1. 如果继续追问底层机制，回答应该深入到什么程度？

不要停在一句结论上，要沿着“语言语义 -> 运行时或编译器机制 -> 工程影响”的顺序回答。

- GC 会从根对象出发标记可达对象，未标记对象在清扫阶段被回收。
- 三色标记和写屏障用于在并发执行用户代码时维持标记正确性。
- `GOGC` 控制下一次 GC 触发时相对上次存活堆大小的增长比例，数值越低 GC 越频繁。
- GC 成本不只来自回收，还来自分配、扫描指针对象、写屏障和 STW 阶段。

面试时可以先用一句话建立主线，再展开关键细节。这样既能让初学者听懂，也能让面试官看到你不是死记硬背。

### 2. 这个知识点在真实项目里怎么取舍？

核心不是知道某个 API 或语法能用，而是知道什么时候该用、什么时候不该用。

- 先用 pprof 和 benchmark 找到主要分配来源，再决定是否优化。
- 减少热点路径临时对象，优先考虑预分配、复用缓冲区和避免不必要装箱。
- 缩短大对象引用生命周期，例如及时置 nil 或复制小片段。
- 不要为了减少 GC 把代码写成难维护的对象池，除非数据证明收益明确。

如果一个选择会影响可读性、性能、并发安全或 API 兼容性，要把这些代价说出来，而不是只给“用 A”或“用 B”的答案。

### 3. 这道题最容易追问哪些坑？

面试官通常会从边界条件和反例继续问，因为这些地方最能区分“会背”和“真懂”。

- 把 GC 当成内存泄漏的根因，忽略仍有引用导致对象可达。
- 频繁手动触发 GC，增加停顿和 CPU 成本。
- 看到堆增长就优化分配，却没有区分正常缓存、存活数据和泄漏。
- 过度使用 `sync.Pool`，导致对象生命周期和状态复用更难控制。

回答这类追问时，最好先指出错误直觉，再解释为什么错，最后给出正确写法或规避方式。

### 4. 如何证明你的判断是对的？

Go 很适合用小实验验证语言语义，也适合用工具验证性能和并发问题。

- 用 `go test -bench . -benchmem` 看分配次数和分配字节数。
- 用 heap profile 看哪些路径分配多、哪些对象存活久。
- 用 `GODEBUG=gctrace=1` 观察 GC 频率、堆大小和停顿信息。

如果问题涉及并发，优先想到 `go test -race`、goroutine profile、block profile 或 trace；如果涉及性能，优先想到 benchmark、`-benchmem`、pprof 和逃逸分析。

### 5. 当规模变大后，这个问题会如何升级？

很多 Go 基础题在小程序里只是语法点，在服务端工程里会变成资源、稳定性和可维护性问题。

- 低 QPS 服务一般不需要围绕 GC 做复杂优化。
- 高 QPS、低延迟服务要把分配率作为核心指标。
- 大堆服务需要关注扫描成本和对象图结构，而不只是总内存。

面试中可以主动补一句规模化后的影响，这会让答案从“语言知识”升级成“工程判断”。

### 6. 初学者应该怎么把这个问题学扎实？

建议按三个层次学习：先写最小可运行例子确认语义，再读官方文档或标准库用法，最后用测试、benchmark 或 profile 观察真实行为。不要只背结论；每个结论都要能回答“为什么”和“在哪些条件下不成立”。
