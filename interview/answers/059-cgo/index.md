# 059. cgo

## 问题

cgo 的成本和使用风险是什么？

## 先给结论

cgo 让 Go 调用 C，也让 C 调用 Go，但它会引入调用开销、构建复杂度、内存所有权和指针传递限制。能不用就不用，需要用时要把边界封装清楚。

## 深入理解

### 1. 这道题真正考察什么

- 是否知道 cgo 调用成本高于普通 Go 函数调用。
- 是否能解释 Go 指针传给 C 的限制。
- 是否理解 C 内存需要手动释放。
- 是否知道 cgo 会影响交叉编译、部署和调试。

### 2. 底层机制要讲清楚

- cgo 需要在 Go 调度器和 C ABI 之间切换，涉及栈、线程和调用约定处理。
- C 代码不受 Go GC 管理，C malloc 的内存必须 C free。
- Go 指针不能随意长期保存在 C 中，否则 GC 无法正确追踪对象生命周期。
- 阻塞 C 调用会影响线程资源，虽然调度器会尽量补充执行 Go 代码的线程。

### 3. 工程实践怎么取舍

- 优先寻找纯 Go 实现，除非必须复用成熟 C 库或系统 API。
- 把 cgo 边界集中在小包里，向业务暴露纯 Go API。
- 减少大量小 cgo 调用，必要时批量传输数据。
- 明确内存由谁分配、谁释放，错误如何转换。

### 4. 常见误区

- 频繁逐条调用 C 函数，调用开销超过实际计算。
- C 分配内存忘记释放，Go pprof 看不到完整问题。
- C 长期保存 Go 指针，违反 cgo pointer rules。
- 忽略目标机器缺少 C 依赖导致部署失败。

## 如何验证理解

- 用 benchmark 比较 cgo 调用粒度对性能的影响。
- 用系统内存工具配合 Go pprof 排查 C 内存泄漏。
- 在 CI 中覆盖目标平台构建，避免本机能编译线上失败。

