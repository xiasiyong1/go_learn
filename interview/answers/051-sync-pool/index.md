# 051. sync.Pool

## 问题

`sync.Pool` 适合什么场景？为什么不能当普通对象池？

## 先给结论

sync.Pool 是临时对象复用工具，目标是降低短生命周期对象分配压力。它可能在 GC 后丢弃对象，不能当作容量稳定、语义可靠的普通对象池。

## 深入理解

### 1. 这道题真正考察什么

- 是否知道 Pool 中对象随时可能被清理。
- 是否能解释 Put 前为什么要 Reset。
- 是否知道 sync.Pool 适合临时、无所有权语义的对象。
- 是否能判断对象池是否真的降低分配而不是增加复杂度。

### 2. 底层机制要讲清楚

- sync.Pool 为每个 P 提供本地缓存，减少竞争。
- GC 周期会影响 Pool 中对象的存留，运行时可以清掉未使用对象。
- Pool 不保证 Get 一定返回之前 Put 的对象，也不保证数量。
- 对象放回 Pool 后，调用方不应再持有和使用它。

### 3. 工程实践怎么取舍

- 适合复用 `bytes.Buffer`、临时大对象、编码缓冲等无业务身份对象。
- Put 前清理状态，避免数据泄露和串扰。
- 对象创建便宜或不在热点路径时，不要引入 Pool。
- 有固定连接、文件句柄、会话语义的资源不要用 sync.Pool 管理。

### 4. 常见误区

- 把 sync.Pool 当连接池或限流池。
- Put 后继续使用对象，造成数据竞争或污染。
- 忘记 Reset，下一次 Get 读到旧数据。
- 对象太小或使用频率低，Pool 开销大于收益。

## 如何验证理解

- 用 benchmark 对比使用 Pool 前后的 allocs/op 和 B/op。
- 用 race detector 检查 Put 后误用。
- 用 heap profile 观察对象复用是否降低分配热点。

## 代码示例

```go
var bufPool = sync.Pool{
	New: func() any {
		return new(bytes.Buffer)
	},
}
```
