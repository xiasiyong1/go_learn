# 056. 锁竞争

## 问题

如何定位和优化锁竞争？

## 先给结论

锁竞争优化的第一步是定位，而不是猜。要弄清哪把锁、谁持有、持有多久、等待多少，以及锁保护的不变量是否能拆分或改造。

## 深入理解

### 1. 这道题真正考察什么

- 是否知道锁竞争会表现为延迟升高、CPU 空等或 goroutine 阻塞。
- 是否会使用 mutex profile、block profile 和 trace。
- 是否能区分减少持锁时间、降低锁频率和拆分锁粒度。
- 是否能评估分片锁、copy-on-write、atomic 等替代方案的代价。

### 2. 底层机制要讲清楚

- 锁竞争来自多个 goroutine 同时需要同一临界区。
- 临界区越长、访问越频繁、热点越集中，竞争越明显。
- RWMutex 在读多写少时可能改善，但读锁本身也有成本，写者也会等待。
- 分片锁通过分散热点降低竞争，但增加遍历、扩容和一致性复杂度。

### 3. 工程实践怎么取舍

- 先通过 profile 找到真实竞争锁。
- 缩短持锁代码，避免持锁 I/O、日志、RPC 或复杂计算。
- 把只读快照和写入路径分离，或用 copy-on-write 降低读锁压力。
- 热点 key 可以分片，跨 key 不变量则不能简单拆锁。

### 4. 常见误区

- 没有 profile 就直接换 RWMutex 或分片锁。
- 拆锁后破坏原本由一把锁保护的整体不变量。
- 持锁调用外部函数，外部函数反过来拿锁导致死锁。
- 只优化平均延迟，忽略 P99 等尾延迟。

## 如何验证理解

- 开启 mutex profile，查看锁等待时间和调用栈。
- 用 block profile 找 channel、select、锁等阻塞点。
- 用压测对比优化前后吞吐和尾延迟。

## 代码示例

```go
runtime.SetMutexProfileFraction(1)
```
