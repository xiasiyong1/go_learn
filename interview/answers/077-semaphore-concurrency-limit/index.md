# 077. 并发限制

## 问题

Go 中如何用信号量限制并发？和 worker pool 有什么区别？

## 先给结论

信号量用于限制同时进入某段逻辑的 goroutine 数量，适合保护下游资源或本地昂贵操作。它和 worker pool 都能限制并发，但表达模型不同：一个限制许可，一个限制任务执行者。

## 深入理解

### 1. 这道题真正考察什么

- 是否知道带缓冲 channel 可以实现简单信号量。
- 是否能解释 acquire 和 release 必须成对。
- 是否能区分并发限制和速率限制。
- 是否知道 context 取消时获取许可不能无限等待。

### 2. 底层机制要讲清楚

- 信号量维护有限许可，获取许可后才能执行受保护逻辑。
- 带缓冲 channel 中写入 token 表示占用许可，读取 token 表示释放许可。
- worker pool 固定执行者数量，任务通过队列分发。
- 加权信号量可以让不同任务消耗不同数量资源。

### 3. 工程实践怎么取舍

- 保护数据库、RPC 下游、文件句柄等有限资源时使用并发限制。
- 用 `defer release()` 确保错误路径释放许可。
- 获取许可时支持 context，避免请求取消后仍等待。
- 如果任务需要排队和生命周期管理，worker pool 可能更清晰。

### 4. 常见误区

- 异常分支忘记释放许可，最终所有请求卡死。
- 把并发限制当速率限制，无法控制单位时间请求数。
- 信号量容量设置过大，保护不了下游。
- 在持有许可期间执行不相关慢操作，降低整体吞吐。

## 如何验证理解

- 写并发测试统计同时执行数量不超过上限。
- 测试任务 panic 或返回错误时许可是否释放。
- 压测不同上限下的延迟、错误率和下游负载。
