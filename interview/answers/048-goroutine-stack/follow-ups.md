# 048. goroutine 栈 - 面试追问

## 追问与参考答案

### 1. 如果继续追问底层机制，回答应该深入到什么程度？

不要停在一句结论上，要沿着“语言语义 -> 运行时或编译器机制 -> 工程影响”的顺序回答。

- goroutine 栈按需增长，空间不足时运行时会分配更大栈并迁移内容。
- 栈上指针需要在迁移后被正确调整，这依赖编译器和运行时的精确信息。
- GC 需要扫描 goroutine 栈上的指针根，活跃 goroutine 越多扫描成本可能越高。
- 阻塞 goroutine 的栈虽然可能很小，但栈上的引用会让堆对象保持可达。

面试时可以先用一句话建立主线，再展开关键细节。这样既能让初学者听懂，也能让面试官看到你不是死记硬背。

### 2. 这个知识点在真实项目里怎么取舍？

核心不是知道某个 API 或语法能用，而是知道什么时候该用、什么时候不该用。

- 可以放心创建 goroutine 处理并发任务，但要有上限和退出路径。
- 深递归算法在 Go 中仍要谨慎，必要时改为迭代。
- 不要把大对象长期挂在 goroutine 栈上等待。
- 服务中监控 goroutine 数量和 goroutine profile。

如果一个选择会影响可读性、性能、并发安全或 API 兼容性，要把这些代价说出来，而不是只给“用 A”或“用 B”的答案。

### 3. 这道题最容易追问哪些坑？

面试官通常会从边界条件和反例继续问，因为这些地方最能区分“会背”和“真懂”。

- 认为 goroutine 轻量就可以无限创建。
- 忽略 goroutine 泄漏导致的堆对象滞留。
- 把递归深度不受控的输入直接交给递归函数。
- 只看堆，不看 goroutine 栈和调度开销。

回答这类追问时，最好先指出错误直觉，再解释为什么错，最后给出正确写法或规避方式。

### 4. 如何证明你的判断是对的？

Go 很适合用小实验验证语言语义，也适合用工具验证性能和并发问题。

- 用 pprof goroutine profile 查看 goroutine 数量和阻塞点。
- 用 trace 观察 goroutine 创建和调度行为。
- 构造深递归测试，观察栈增长和性能变化。

如果问题涉及并发，优先想到 `go test -race`、goroutine profile、block profile 或 trace；如果涉及性能，优先想到 benchmark、`-benchmem`、pprof 和逃逸分析。

### 5. 当规模变大后，这个问题会如何升级？

很多 Go 基础题在小程序里只是语法点，在服务端工程里会变成资源、稳定性和可维护性问题。

- 小并发场景 goroutine 成本很低。
- 百万级 goroutine 或大量泄漏时，栈和调度成本会非常明显。
- 高并发服务要用 worker pool、连接池和 backpressure 控制数量。

面试中可以主动补一句规模化后的影响，这会让答案从“语言知识”升级成“工程判断”。

### 6. 初学者应该怎么把这个问题学扎实？

建议按三个层次学习：先写最小可运行例子确认语义，再读官方文档或标准库用法，最后用测试、benchmark 或 profile 观察真实行为。不要只背结论；每个结论都要能回答“为什么”和“在哪些条件下不成立”。
