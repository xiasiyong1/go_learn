# 048. goroutine 栈

## 问题

goroutine 栈如何增长？为什么 goroutine 可以很轻量？

## 先给结论

goroutine 轻量的重要原因之一是初始栈很小，并且可以按需增长和收缩。它不是免费资源，数量过多仍会带来内存、调度和 GC 扫描成本。

## 深入理解

### 1. 这道题真正考察什么

- 是否知道 goroutine 初始栈远小于传统线程栈。
- 是否能解释栈增长为什么需要复制和指针调整。
- 是否知道深递归和大量 goroutine 会增加栈内存压力。
- 是否能说明 goroutine 泄漏会连带持有栈引用对象。

### 2. 底层机制要讲清楚

- goroutine 栈按需增长，空间不足时运行时会分配更大栈并迁移内容。
- 栈上指针需要在迁移后被正确调整，这依赖编译器和运行时的精确信息。
- GC 需要扫描 goroutine 栈上的指针根，活跃 goroutine 越多扫描成本可能越高。
- 阻塞 goroutine 的栈虽然可能很小，但栈上的引用会让堆对象保持可达。

### 3. 工程实践怎么取舍

- 可以放心创建 goroutine 处理并发任务，但要有上限和退出路径。
- 深递归算法在 Go 中仍要谨慎，必要时改为迭代。
- 不要把大对象长期挂在 goroutine 栈上等待。
- 服务中监控 goroutine 数量和 goroutine profile。

### 4. 常见误区

- 认为 goroutine 轻量就可以无限创建。
- 忽略 goroutine 泄漏导致的堆对象滞留。
- 把递归深度不受控的输入直接交给递归函数。
- 只看堆，不看 goroutine 栈和调度开销。

## 如何验证理解

- 用 pprof goroutine profile 查看 goroutine 数量和阻塞点。
- 用 trace 观察 goroutine 创建和调度行为。
- 构造深递归测试，观察栈增长和性能变化。

## 代码示例

```go
go func() {
	doWork()
}()
```
