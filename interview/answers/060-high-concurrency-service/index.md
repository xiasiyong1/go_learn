# 060. 高并发 Go 服务设计

## 问题

如何设计高并发 Go 服务的整体稳定性方案？

## 先给结论

高并发 Go 服务设计不是单点技巧，而是容量评估、资源隔离、超时取消、限流熔断、缓存、并发模型、可观测性和压测验证的组合。

## 深入理解

### 1. 这道题真正考察什么

- 是否能先定义目标：QPS、延迟、错误率、资源预算和降级策略。
- 是否能识别瓶颈在 CPU、内存、锁、网络、数据库还是下游服务。
- 是否能设计超时、限流、熔断、重试和背压。
- 是否能用指标、日志、trace、pprof 和压测闭环验证。

### 2. 底层机制要讲清楚

- Go 能轻松创建并发，但系统容量仍受 CPU、内存、fd、连接池、锁和下游限额约束。
- context 和 deadline 防止请求在调用链中无限堆积。
- 限流和并发控制保护本服务，熔断和降级保护调用链。
- 缓存和 singleflight 可以减少下游压力，但要处理一致性、失效和击穿。

### 3. 工程实践怎么取舍

- 入口层设置请求大小、超时、限流和认证成本上限。
- 业务层控制 goroutine 数量、队列长度和下游并发。
- 依赖层设置连接池、超时、重试退避和熔断。
- 上线前用压测和故障注入验证容量、降级和恢复。

### 4. 常见误区

- 只关注 QPS，不关注 P99、错误率和资源峰值。
- 无限 goroutine 加无限队列，短期吞吐高，故障时雪崩。
- 无脑重试，放大下游故障。
- 没有可观测性，出问题后不知道瓶颈在哪。

## 如何验证理解

- 建立 RED/USE 指标：请求率、错误率、延迟、资源使用、饱和度。
- 用 pprof 定位 CPU、内存和锁竞争。
- 用 trace 分析调度、阻塞和 GC 影响。
- 压测正常、突发、下游慢、下游错误和部分依赖不可用场景。

## 代码示例

```go
ctx, cancel := context.WithTimeout(r.Context(), 300*time.Millisecond)
defer cancel()
```
