# 049. 三色标记和写屏障

## 问题

三色标记、写屏障和 STW 分别是什么？

## 先给结论

三色标记是 GC 并发标记的抽象，写屏障用于在用户代码并发修改对象图时保持标记正确性，STW 则用于必须全局一致的短暂阶段。

## 深入理解

### 1. 这道题真正考察什么

- 是否能解释白、灰、黑三类对象的含义。
- 是否知道并发标记时对象引用关系会被用户代码改变。
- 是否能说明写屏障为什么存在。
- 是否理解低延迟 GC 不等于无成本。

### 2. 底层机制要讲清楚

- 白色表示未发现可达，灰色表示已发现但其引用还没扫描完，黑色表示已扫描完。
- 并发标记中，如果黑色对象新增指向白色对象且没有屏障，白色对象可能被错误回收。
- 写屏障在指针写入时执行额外逻辑，帮助 GC 维护三色不变式。
- STW 用于根扫描准备、标记终止等需要一致视图的阶段，Go 尽量缩短这些阶段。

### 3. 工程实践怎么取舍

- 业务代码一般不直接操作写屏障，但应理解指针写入和对象图复杂度会影响 GC。
- 减少不必要指针对象和长生命周期对象，可以降低扫描压力。
- 性能分析时关注 GC CPU、堆大小、分配率和暂停时间，而不是只看 STW。
- 不要手动调 GC 解决对象仍可达导致的内存问题。

### 4. 常见误区

- 以为三色标记是 Go 独有实现细节，不需要理解可达性。
- 认为有写屏障就没有 GC 成本。
- 只关注 STW，忽略并发 GC 占用 CPU。
- 把内存不下降归因于 GC 不工作，而不是对象仍可达。

## 如何验证理解

- 用 `GODEBUG=gctrace=1` 观察 GC 周期、暂停和堆大小。
- 用 heap profile 找存活对象引用链。
- 通过 benchmark 观察减少指针对象后 GC 指标是否改善。

## 代码示例

```bash
GODEBUG=gctrace=1 go test ./...
```
