# 058. unsafe

## 问题

`unsafe` 能做什么？使用边界是什么？

## 先给结论

unsafe 允许绕过 Go 类型系统做指针转换、内存布局访问等操作。它能解决极少数性能或系统编程问题，但代价是失去类型安全、可移植性和 GC 保护边界。

## 深入理解

### 1. 这道题真正考察什么

- 是否知道 `unsafe.Pointer` 可以在不同指针类型间转换。
- 是否能区分 `unsafe.Pointer` 和 `uintptr`。
- 是否理解 uintptr 不是指针，不能保证对象存活。
- 是否知道 unsafe 代码可能依赖内存布局和 Go 版本实现。

### 2. 底层机制要讲清楚

- `unsafe.Pointer` 是指针转换的中介，GC 仍能把它识别为指针。
- `uintptr` 是整数，GC 不把它当引用，转换后对象可能被移动或回收风险要考虑。
- 结构体字段布局、对齐和填充会影响 unsafe 偏移访问。
- 字符串和切片的零拷贝转换可能破坏不可变性或生命周期假设。

### 3. 工程实践怎么取舍

- 只有在 profiling 证明必要，且安全边界能被封装和测试时才用 unsafe。
- 把 unsafe 限制在很小的内部包，不向外泄露不安全指针。
- 优先使用标准库提供的安全 API，例如 Go 新版本中的转换辅助能力。
- 写清楚对象生命周期、对齐、别名和不可变性假设。

### 4. 常见误区

- 把 uintptr 长期保存后再转回指针。
- 把 []byte 零拷贝转 string 后继续修改 byte。
- 依赖结构体私有布局，Go 版本或架构变化后崩溃。
- 为了微小优化引入难以审计的内存安全风险。

## 如何验证理解

- 为 unsafe 包写跨架构、race、fuzz 和边界测试。
- 用 benchmark 证明 unsafe 版本确有收益。
- 代码 review 时逐条检查生命周期和 GC 可见性。

## 代码示例

```go
size := unsafe.Sizeof(MyStruct{})
```
