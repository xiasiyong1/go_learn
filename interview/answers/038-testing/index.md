# 038. 测试

## 问题

Go 表格驱动测试、子测试和 mock 应该怎么写？

## 先给结论

Go 测试强调简单、靠近代码和工具链内置。表格驱动测试、子测试、mock 和 benchmark 的重点是覆盖行为边界，而不是追求形式完整。

## 深入理解

### 1. 这道题真正考察什么

- 是否能写出可读的表格驱动测试。
- 是否知道子测试可以用 `t.Run` 分组并独立运行。
- 是否理解 `t.Helper()` 能让失败行号指向调用者。
- 是否能区分单元测试、集成测试、benchmark 和 fuzz test。

### 2. 底层机制要讲清楚

- 表格驱动测试把输入、期望和名称放在数据表里，减少重复逻辑。
- 子测试让每个 case 有独立名称，便于筛选和并行。
- mock 的本质是替换外部依赖，让测试聚焦当前单元行为。
- benchmark 默认关注性能，需要配合 `-benchmem` 看分配。

### 3. 工程实践怎么取舍

- 优先测试可观察行为，不测试内部实现细节。
- 外部依赖用接口、httptest、临时目录或测试容器隔离。
- 对错误路径、边界值、并发取消和序列化兼容性补充测试。
- mock 不要过度指定调用细节，否则重构会让测试脆弱。

### 4. 常见误区

- 只测试 happy path，错误路径无人覆盖。
- 测试共享全局状态，导致用例顺序相关。
- 并行子测试捕获循环变量出错。
- benchmark 没有阻止编译器优化，结果不可信。

## 如何验证理解

- 用 `go test ./...` 跑完整测试。
- 用 `go test -run TestName/subcase` 定位子测试。
- 用 `go test -race` 跑并发相关测试。
- 用 `go test -bench . -benchmem` 验证性能变化。

## 代码示例

```go
for _, tt := range tests {
	t.Run(tt.name, func(t *testing.T) {
		got := Add(tt.a, tt.b)
		if got != tt.want {
			t.Fatalf("got %d, want %d", got, tt.want)
		}
	})
}
```
