# 044. map 底层实现

## 问题

Go map 底层大致如何实现？扩容时发生了什么？

## 先给结论

Go map 底层是哈希表，使用 bucket、溢出 bucket、哈希种子和渐进扩容等机制。理解它能解释无序遍历、元素不可取地址和并发不安全。

## 深入理解

### 1. 这道题真正考察什么

- 是否知道 map 查找依赖哈希和相等比较。
- 是否能解释 bucket 和溢出 bucket 的大致结构。
- 是否知道扩容可能渐进迁移数据。
- 是否能说明 map 元素为什么不能直接取地址。

### 2. 底层机制要讲清楚

- 每个 bucket 存放多个 key/value 及部分哈希信息。
- 哈希冲突会使用溢出 bucket 链接。
- 装载因子过高或溢出 bucket 过多时会触发扩容或整理。
- 扩容和迁移可能移动元素位置，因此暴露元素地址会破坏安全性。

### 3. 工程实践怎么取舍

- 预估元素数量时用 `make(map[K]V, n)` 减少扩容。
- 需要修改 map 中结构体字段时，取出副本、修改后再写回，或存指针值。
- 需要稳定顺序时不要依赖 map，显式维护有序结构。
- 高并发访问必须加锁、使用 sync.Map 或改变所有权模型。

### 4. 常见误区

- 对 `m[k].Field` 直接赋值，编译失败后不理解原因。
- 认为 map 遍历无序只是偶然，写出依赖顺序的代码。
- 大量写入不预分配，频繁扩容影响性能。
- 使用可变对象内容作为 key，却没有保证 key 不会变化。

## 如何验证理解

- 用 benchmark 比较预分配和不预分配的写入性能。
- 写小例子验证 map 元素不可取地址和结构体字段不可直接改。
- 用 pprof 观察大 map 场景的哈希和分配成本。

## 代码示例

```go
m := make(map[string]int, 1024) // 合理预估容量可减少扩容
```
