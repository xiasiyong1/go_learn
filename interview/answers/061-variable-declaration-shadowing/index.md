# 061. 变量

## 问题

Go 中 `var`、短变量声明和变量遮蔽应该怎么理解？

## 先给结论

`var` 更适合包级变量、零值声明和需要显式类型的场景，`:=` 只能用于函数内部并且至少声明一个新变量。变量遮蔽不是语法错误，但会让读者误以为修改的是外层变量，是 Go 初学者很常见的逻辑坑。

## 深入理解

### 1. 这道题真正考察什么

- 是否知道 `:=` 不能用于包级作用域。
- 是否能解释短变量声明中“至少有一个新变量”的规则。
- 是否能识别内层作用域重新声明导致的变量遮蔽。
- 是否知道零值声明能减少无意义初始化。

### 2. 底层机制要讲清楚

- Go 的变量作用域由代码块决定，内层块可以声明与外层同名的变量。
- `:=` 是声明，不是单纯赋值；如果左侧没有任何新变量，编译器会报错。
- 短变量声明可以同时复用旧变量和声明新变量，例如 `v, err := f()` 中 err 可能是新变量，也可能被遮蔽。
- 包级变量初始化发生在 init 之前，并且应避免复杂副作用。

### 3. 工程实践怎么取舍

- 局部临时变量优先用 `:=`，让代码更短。
- 需要明确类型、零值或包级声明时使用 `var`。
- 错误处理里避免在内层 `if` 或 `for` 中意外遮蔽外层 `err`。
- 代码块较长时，不要复用过于通用的变量名。

### 4. 常见误区

- 在 `if v, err := f(); err != nil { ... }` 外部使用不到 v 或 err。
- 内层 `x := x + 1` 创建了新 x，外层 x 没有变化。
- 把短变量声明当成赋值，结果编译或逻辑不符合预期。
- 包级变量承担复杂初始化，导致测试和依赖顺序难以控制。

## 如何验证理解

- 写小例子打印内外层变量地址和值，观察遮蔽效果。
- 用 `go vet` 或静态检查工具发现可疑 shadow 场景。
- 在 code review 中重点检查 `:=` 出现在错误处理和嵌套代码块的位置。
