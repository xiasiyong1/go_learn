# 001. 数组和切片 - 面试追问

## 追问与参考答案

### 1. 如果继续追问底层机制，回答应该深入到什么程度？

不要停在一句结论上，要沿着“语言语义 -> 运行时或编译器机制 -> 工程影响”的顺序回答。

- 切片头可以理解为三元组：指向底层数组的指针、长度、容量。复制切片只复制这三个字段。
- `append` 先看容量是否足够；容量足够时原地写入，容量不足时分配新数组并复制旧元素。
- 切片扩容策略是运行时实现细节，不能依赖固定倍数写业务逻辑；只应该依赖“可能换底层数组”这个语义。
- 数组长度是类型的一部分，所以 `[3]int` 和 `[4]int` 不能直接赋值。这个设计让数组更适合表达固定大小的数据。

面试时可以先用一句话建立主线，再展开关键细节。这样既能让初学者听懂，也能让面试官看到你不是死记硬背。

### 2. 这个知识点在真实项目里怎么取舍？

核心不是知道某个 API 或语法能用，而是知道什么时候该用、什么时候不该用。

- 函数只读少量元素时传切片；需要固定大小和完整拷贝语义时才考虑数组。
- 要保留子切片但释放原大数组时，用 `copy` 或 `append([]T(nil), src...)` 复制需要的部分。
- 需要防止追加污染原数组时，可以使用完整切片表达式限制容量，例如 `s := old[:n:n]`。
- 性能敏感路径提前估算容量，用 `make([]T, 0, n)` 减少扩容和复制。

如果一个选择会影响可读性、性能、并发安全或 API 兼容性，要把这些代价说出来，而不是只给“用 A”或“用 B”的答案。

### 3. 这道题最容易追问哪些坑？

面试官通常会从边界条件和反例继续问，因为这些地方最能区分“会背”和“真懂”。

- 误以为切片赋值会复制元素，导致一个地方修改影响另一个地方。
- 误以为 `append` 一定生成新数组，结果容量够时覆盖了共享底层数组中的后续元素。
- 只截取小片段返回，忘记复制，导致大对象迟迟不能被回收。
- 把扩容倍数当成语言规范，写出依赖运行时版本的代码。

回答这类追问时，最好先指出错误直觉，再解释为什么错，最后给出正确写法或规避方式。

### 4. 如何证明你的判断是对的？

Go 很适合用小实验验证语言语义，也适合用工具验证性能和并发问题。

- 打印 `len`、`cap` 和元素变化，观察共享底层数组导致的写入传播。
- 用 `go test -bench . -benchmem` 比较预分配和不预分配的分配次数。
- 用 `pprof` 或内存测试构造大切片截取场景，确认复制后大数组可以释放。

如果问题涉及并发，优先想到 `go test -race`、goroutine profile、block profile 或 trace；如果涉及性能，优先想到 benchmark、`-benchmem`、pprof 和逃逸分析。

### 5. 当规模变大后，这个问题会如何升级？

很多 Go 基础题在小程序里只是语法点，在服务端工程里会变成资源、稳定性和可维护性问题。

- 数据量很小时，清晰性通常比微优化更重要。
- 数据量很大或生命周期很长时，要特别关注切片是否意外持有大数组。
- 高并发场景里，切片共享不仅是内存问题，还可能变成数据竞争问题。

面试中可以主动补一句规模化后的影响，这会让答案从“语言知识”升级成“工程判断”。

### 6. 初学者应该怎么把这个问题学扎实？

建议按三个层次学习：先写最小可运行例子确认语义，再读官方文档或标准库用法，最后用测试、benchmark 或 profile 观察真实行为。不要只背结论；每个结论都要能回答“为什么”和“在哪些条件下不成立”。
