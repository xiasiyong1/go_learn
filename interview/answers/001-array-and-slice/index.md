# 001. 数组和切片

## 问题

Go 中数组和切片有什么区别？切片扩容时需要注意什么？

## 先给结论

数组强调固定长度和值语义，切片强调对底层数组的动态视图。面试不能只说“数组定长、切片变长”，还要说清楚切片头、共享底层数组、容量、扩容和内存滞留之间的关系。

## 深入理解

### 1. 这道题真正考察什么

- 能否区分值拷贝和引用共享：数组传参复制整个值，切片传参复制的是切片头。
- 能否理解 `len` 和 `cap` 的边界：`len` 决定可访问范围，`cap` 决定还能在当前底层数组上追加多少元素。
- 能否预判 `append` 是否会影响其他切片：关键不是变量名，而是是否仍共享同一个底层数组。
- 能否解释截取大切片后的内存滞留：小切片活着时，底层大数组仍被 GC 认为可达。

### 2. 底层机制要讲清楚

- 切片头可以理解为三元组：指向底层数组的指针、长度、容量。复制切片只复制这三个字段。
- `append` 先看容量是否足够；容量足够时原地写入，容量不足时分配新数组并复制旧元素。
- 切片扩容策略是运行时实现细节，不能依赖固定倍数写业务逻辑；只应该依赖“可能换底层数组”这个语义。
- 数组长度是类型的一部分，所以 `[3]int` 和 `[4]int` 不能直接赋值。这个设计让数组更适合表达固定大小的数据。

### 3. 工程实践怎么取舍

- 函数只读少量元素时传切片；需要固定大小和完整拷贝语义时才考虑数组。
- 要保留子切片但释放原大数组时，用 `copy` 或 `append([]T(nil), src...)` 复制需要的部分。
- 需要防止追加污染原数组时，可以使用完整切片表达式限制容量，例如 `s := old[:n:n]`。
- 性能敏感路径提前估算容量，用 `make([]T, 0, n)` 减少扩容和复制。

### 4. 常见误区

- 误以为切片赋值会复制元素，导致一个地方修改影响另一个地方。
- 误以为 `append` 一定生成新数组，结果容量够时覆盖了共享底层数组中的后续元素。
- 只截取小片段返回，忘记复制，导致大对象迟迟不能被回收。
- 把扩容倍数当成语言规范，写出依赖运行时版本的代码。

## 如何验证理解

- 打印 `len`、`cap` 和元素变化，观察共享底层数组导致的写入传播。
- 用 `go test -bench . -benchmem` 比较预分配和不预分配的分配次数。
- 用 `pprof` 或内存测试构造大切片截取场景，确认复制后大数组可以释放。

## 代码示例

```go
package main

import "fmt"

func main() {
	a := []int{1, 2, 3}
	b := a[:2]

	b = append(b, 100)

	fmt.Println(a) // [1 2 100]
	fmt.Println(b) // [1 2 100]
}
```
