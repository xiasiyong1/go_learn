# 025. 切片内存滞留

## 问题

大切片截取小切片为什么可能导致内存不释放？

## 先给结论

大切片截取小切片后内存不释放，是因为小切片仍指向原底层数组。GC 只看可达性，不知道你逻辑上只需要其中一小段。

## 深入理解

### 1. 这道题真正考察什么

- 是否能用切片头解释小切片为什么持有大数组。
- 是否知道 `cap` 可能暴露原数组后续区域。
- 是否能给出复制解除引用的写法。
- 是否能识别字符串子串、bytes buffer 等类似持有问题。

### 2. 底层机制要讲清楚

- 切片头的数据指针通常指向底层数组中的某个元素。
- 只要切片头可达，数据指针指向的底层数组就可达，整个数组不能被回收。
- 完整切片表达式 `s[:len:len]` 可以限制容量，但不能释放原数组；释放需要复制。
- 复制后新切片指向新的小数组，原大数组如果无其他引用就可以回收。

### 3. 工程实践怎么取舍

- 从大文件、大响应、大缓冲中只保留少量数据时，一定复制需要的片段。
- 返回子切片前考虑是否要限制 cap，避免调用方 append 污染原数组。
- 对长期缓存的数据，宁可多一次复制，也不要长期持有大内存。
- 热路径中复制也有成本，要结合生命周期和大小判断。

### 4. 常见误区

- 以为 `s = s[:10]` 会缩小底层数组。
- 只使用完整切片表达式限制 cap，却以为可以释放大数组。
- 缓存小字段时无意持有整条大消息。
- 用 pprof 只看分配点，不看对象为什么仍然存活。

## 如何验证理解

- 构造大切片截取小切片，用 heap profile 观察复制前后的存活内存。
- 打印 `len` 和 `cap` 检查子切片是否暴露过大容量。
- 为解析函数写内存回归 benchmark，防止重新引入持有问题。

## 代码示例

```go
small := append([]byte(nil), big[:10]...)
big = nil
```
