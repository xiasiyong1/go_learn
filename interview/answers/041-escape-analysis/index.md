# 041. 逃逸分析

## 问题

什么是逃逸分析？如何判断变量分配在栈上还是堆上？

## 先给结论

逃逸分析是编译器判断变量生命周期是否超出当前栈帧的过程。逃逸到堆不是语义错误，但会增加分配和 GC 成本。

## 深入理解

### 1. 这道题真正考察什么

- 是否知道返回局部变量指针在 Go 中合法。
- 是否能解释变量为什么可能从栈分配变成堆分配。
- 是否知道接口装箱、闭包捕获、跨 goroutine 使用都可能导致逃逸。
- 是否能用编译器输出验证，而不是靠猜。

### 2. 底层机制要讲清楚

- 栈分配生命周期随函数调用结束，成本低且不需要 GC 单独管理。
- 如果编译器无法证明对象不会在函数返回后继续使用，就必须放到堆上。
- 逃逸分析发生在编译期，目标是保持语义正确和尽量减少堆分配。
- 是否逃逸受代码形态、内联、接口调用和编译器版本影响。

### 3. 工程实践怎么取舍

- 性能敏感路径减少不必要指针返回、接口装箱和闭包捕获。
- 不要为了避免逃逸牺牲清晰 API，先用 benchmark 证明问题。
- 大对象频繁逃逸时，考虑改变数据结构、调用边界或复用策略。
- 读取 `-gcflags=-m` 输出时关注热点函数，而不是所有 escape 提示。

### 4. 常见误区

- 认为“返回局部变量指针”会变成悬空指针。
- 认为使用指针一定减少拷贝且更快。
- 看到 escape 就盲目优化，破坏代码可读性。
- 忽略接口参数导致的装箱和逃逸。

## 如何验证理解

- 用 `go build -gcflags='-m -m'` 查看逃逸原因。
- 用 benchmark 的 `allocs/op` 验证改动是否减少分配。
- 用 pprof heap profile 确认逃逸对象是否真正占热点。

## 代码示例

```bash
go build -gcflags="-m" ./...
```
