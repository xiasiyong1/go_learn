# 047. 内存分配器

## 问题

Go 内存分配器大致怎么工作？小对象和大对象有什么区别？

## 先给结论

Go 内存分配器按对象大小采用不同策略，小对象通过 size class、mcache、mcentral、mheap 等层级快速分配，大对象直接从堆管理结构获取。

## 深入理解

### 1. 这道题真正考察什么

- 是否知道小对象和大对象分配路径不同。
- 是否能解释 size class 的作用。
- 是否理解分配器和 GC 是协同关系。
- 是否知道减少小对象分配能降低分配成本和 GC 扫描压力。

### 2. 底层机制要讲清楚

- 小对象会被归入固定 size class，减少碎片并提升复用效率。
- 每个 P 有 mcache，可在无锁或低锁竞争情况下快速分配。
- mcentral 管理某个 size class 的 span，mheap 管理更大粒度的堆内存。
- 大对象通常按页级别直接从堆分配，成本和碎片影响更明显。

### 3. 工程实践怎么取舍

- 热点路径减少临时对象和指针对象数量。
- 预分配 slice/map，避免反复扩容。
- 复用 buffer 时要先证明分配是瓶颈，并处理好状态清理。
- 关注对象是否含指针，含指针对象会增加 GC 扫描成本。

### 4. 常见误区

- 认为所有分配成本一样。
- 为了复用对象引入复杂对象池，状态未清理导致数据串扰。
- 只看 allocs/op，不看对象是否长期存活和扫描成本。
- 用 unsafe 绕过分配器，破坏 GC 安全。

## 如何验证理解

- 用 benchmark 的 `B/op` 和 `allocs/op` 观察分配。
- 用 heap profile 看分配热点和存活对象。
- 用 `GODEBUG=gctrace=1` 观察分配变化对 GC 的影响。

## 代码示例

```go
buf := make([]byte, 0, 4096) // 预分配可减少重复扩容
```
