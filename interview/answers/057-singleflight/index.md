# 057. singleflight

## 问题

`singleflight` 解决什么问题？适合哪些场景？

## 先给结论

singleflight 用于合并同一 key 的并发重复请求，让只有一个实际执行，其他请求等待并共享结果。它解决缓存击穿和重复计算，不等于缓存本身。

## 深入理解

### 1. 这道题真正考察什么

- 是否知道 singleflight 合并的是同一 key 的并发请求。
- 是否能区分 singleflight、mutex 和 cache。
- 是否知道慢请求或失败结果会影响所有等待者。
- 是否能处理 context、超时和 key 粒度。

### 2. 底层机制要讲清楚

- Group 记录正在执行的 key，后续同 key 请求挂到同一个调用上。
- 第一个调用完成后，结果和错误会返回给所有等待者。
- singleflight 不保存长期结果，请求结束后 key 状态会移除。
- 它降低并发重复工作，但不能限制不同 key 的总并发。

### 3. 工程实践怎么取舍

- 缓存 miss 后加载数据库或远端配置时适合使用。
- key 要足够精确，避免无关请求互相等待。
- 执行函数要有 timeout，避免所有等待者被慢调用拖住。
- 失败是否共享、是否重试、是否缓存负结果，要结合业务设计。

### 4. 常见误区

- 把 singleflight 当缓存，结果每次非并发请求仍然执行。
- key 设计过粗，造成不必要串行化。
- 执行函数很慢且无超时，大量请求堆积等待。
- 忽视错误共享，导致一次下游失败影响所有同 key 请求。

## 如何验证理解

- 写并发测试，确认同 key 多请求只执行一次函数。
- 测试不同 key 是否并发执行。
- 故障注入慢调用和错误，验证超时和错误传播策略。

## 代码示例

```go
var g singleflight.Group
v, err, shared := g.Do(key, func() (any, error) {
	return loadFromDB(key)
})
_ = shared
```
