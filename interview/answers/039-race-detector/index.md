# 039. race detector

## 问题

Go race detector 能发现什么？不能发现什么？

## 先给结论

race detector 能在运行时发现数据竞争：两个 goroutine 访问同一内存位置，至少一个是写，并且没有同步。它不能证明程序没有业务竞态，也不能覆盖未执行路径。

## 深入理解

### 1. 这道题真正考察什么

- 是否能区分数据竞争和竞态条件。
- 是否知道 race detector 需要运行相关代码路径才可能发现问题。
- 是否理解 race 检测会让程序变慢、内存变大。
- 是否能根据报告堆栈定位读写双方。

### 2. 底层机制要讲清楚

- race detector 对程序进行插桩，记录内存访问和同步事件。
- 它基于 happens-before 关系判断两个访问是否缺少同步。
- 锁、channel、atomic 等同步原语能建立 happens-before。
- 没有报 race 只能说明当前执行覆盖和时序下没发现。

### 3. 工程实践怎么取舍

- 并发代码、测试和 CI 中定期运行 `go test -race`。
- 看到 race 报告要修同步模型，而不是靠 sleep 或 recover。
- 业务竞态需要额外的状态机测试或并发压力测试。
- 性能测试不要直接使用 race 结果衡量真实性能。

### 4. 常见误区

- 把业务结果顺序错误当成 race detector 一定能发现的问题。
- 只在单元测试里跑 race，但测试没有覆盖并发路径。
- 使用 atomic 写普通读，race 仍可能报告或逻辑仍不安全。
- 忽略 race 报告中的创建 goroutine 堆栈。

## 如何验证理解

- 用 `go test -race ./...` 扫描测试覆盖路径。
- 对服务可构建 `-race` 版本跑集成或压测环境。
- 根据报告中的 Read/Write 堆栈找到共享变量和缺失同步点。

## 代码示例

```bash
go test -race ./...
go run -race main.go
```
