# 039. race detector - 面试追问

## 追问与参考答案

### 1. 如果继续追问底层机制，回答应该深入到什么程度？

不要停在一句结论上，要沿着“语言语义 -> 运行时或编译器机制 -> 工程影响”的顺序回答。

- race detector 对程序进行插桩，记录内存访问和同步事件。
- 它基于 happens-before 关系判断两个访问是否缺少同步。
- 锁、channel、atomic 等同步原语能建立 happens-before。
- 没有报 race 只能说明当前执行覆盖和时序下没发现。

面试时可以先用一句话建立主线，再展开关键细节。这样既能让初学者听懂，也能让面试官看到你不是死记硬背。

### 2. 这个知识点在真实项目里怎么取舍？

核心不是知道某个 API 或语法能用，而是知道什么时候该用、什么时候不该用。

- 并发代码、测试和 CI 中定期运行 `go test -race`。
- 看到 race 报告要修同步模型，而不是靠 sleep 或 recover。
- 业务竞态需要额外的状态机测试或并发压力测试。
- 性能测试不要直接使用 race 结果衡量真实性能。

如果一个选择会影响可读性、性能、并发安全或 API 兼容性，要把这些代价说出来，而不是只给“用 A”或“用 B”的答案。

### 3. 这道题最容易追问哪些坑？

面试官通常会从边界条件和反例继续问，因为这些地方最能区分“会背”和“真懂”。

- 把业务结果顺序错误当成 race detector 一定能发现的问题。
- 只在单元测试里跑 race，但测试没有覆盖并发路径。
- 使用 atomic 写普通读，race 仍可能报告或逻辑仍不安全。
- 忽略 race 报告中的创建 goroutine 堆栈。

回答这类追问时，最好先指出错误直觉，再解释为什么错，最后给出正确写法或规避方式。

### 4. 如何证明你的判断是对的？

Go 很适合用小实验验证语言语义，也适合用工具验证性能和并发问题。

- 用 `go test -race ./...` 扫描测试覆盖路径。
- 对服务可构建 `-race` 版本跑集成或压测环境。
- 根据报告中的 Read/Write 堆栈找到共享变量和缺失同步点。

如果问题涉及并发，优先想到 `go test -race`、goroutine profile、block profile 或 trace；如果涉及性能，优先想到 benchmark、`-benchmem`、pprof 和逃逸分析。

### 5. 当规模变大后，这个问题会如何升级？

很多 Go 基础题在小程序里只是语法点，在服务端工程里会变成资源、稳定性和可维护性问题。

- 小并发代码 race detector 很有效。
- 大型系统中需要结合压测、trace、日志和代码审查补足未覆盖路径。
- race detector 是发现工具，不是并发设计正确性的证明。

面试中可以主动补一句规模化后的影响，这会让答案从“语言知识”升级成“工程判断”。

### 6. 初学者应该怎么把这个问题学扎实？

建议按三个层次学习：先写最小可运行例子确认语义，再读官方文档或标准库用法，最后用测试、benchmark 或 profile 观察真实行为。不要只背结论；每个结论都要能回答“为什么”和“在哪些条件下不成立”。
